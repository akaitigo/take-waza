# ADR-001: 竹ひご編みパターンのグラフ構造設計

## ステータス

承認済み (2026-03-30)

## コンテキスト

竹細工の編みパターンをCADツール上でモデル化するために、データ構造の設計が必要。
竹ひご同士の交差関係（上下）、位置、パラメータ（幅・厚さ）を表現し、
3Dレンダリングとステップバイステップ図解生成の両方に対応する必要がある。

検討した選択肢:
1. **隣接行列ベースのグラフ** - 密なグラフ向き。竹細工パターンは疎なので非効率
2. **隣接リストベースのグラフ（petgraph crate使用）** - 汎用的だがWASM互換性に懸念
3. **カスタムノード・エッジ構造体** - ドメイン特化で最適化可能。WASM互換性を完全に制御

## 決定

**選択肢3: カスタムノード・エッジ構造体** を採用する。

### 構造

- `Node`: 交差点。位置座標 (x, y, z)、交差タイプ（OVER/UNDER）
- `Edge`: 竹ひご。ノードペア接続、幅、厚さ、素材タイプ
- `Pattern`: パターン定義。名前、ユニットセル定義、タイリング規則
- `WeavingGraph`: ノード・エッジの集合 + パターンメタデータ

### パターン生成方式

- ユニットセル（繰り返し単位）をパラメトリックに定義
- タイリング: 行列のオフセット計算でユニットセルを繰り返し配置
- パラメータ変更時はグラフ全体を再生成（差分更新はMVP後に検討）

### シリアライゼーション

- serde による JSON シリアライゼーション/デシリアライゼーション
- wasm-bindgen で TypeScript 側に `JsValue` として公開

## 根拠

- petgraph はWASMビルドで動作するが、竹細工ドメインに不要な汎用性がバイナリサイズを増大させる
- カスタム構造体なら serde + wasm-bindgen で直接 JSON 変換でき、FFI が簡潔
- MVP段階のパターン規模（10-100ノード）ではパフォーマンス差は無視できる
- 3Dレンダリング用のデータ（位置・交差タイプ）をノードに直接持たせることで変換コストを削減

## リスク

- 大規模パターン（1000ノード超）でグラフアルゴリズムの自前実装がボトルネックになる可能性
  - 緩和策: Phase 1 で petgraph への移行パスを確保（インターフェースを安定させる）
- ステップ分解（トポロジカルソート）の自前実装にバグが入るリスク
  - 緩和策: テストケースで既知パターンの正しい編み順序を検証
